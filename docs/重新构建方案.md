# 重新构建方案

## 一、岗位画像专家节点 (Job Expert)重构

### 详细设计

* 1.对于岗位画像专家节点 (Job Expert)，我们直接取消sql数据库的调用逻辑，直接在知识库中检索岗位画像即可。

* 2.知识库重构，岗位画像的json字段如下：

| 字段名 | 子字段 | 类型 | 说明 | 示例 |  
| :--- | :--- | :--- | :--- | :--- |
|**name**| - |string |岗位类型名称|C_C++| 
| **summary** | - | string | **岗位综述**：对该岗位的职责核心、行业地位及主要工作内容的宏观总结。 | 该岗位是软件开发的核心力量，负责... |  
| **skills** | `name` | string | **专业技能要求**：该岗位必须掌握的硬技能、工具、框架或行业知识。 | Java开发、市场数据分析 |  
| | `evidence` | string | **要求摘要**：从原始数据中提取的技能应用场景或详细要求描述。 | 精通SpringBoot框架，具备微服务架构设计能力。 |  
| **thresholds** | `name` | string | **基础门槛要求**：入职该岗位的刚性过滤条件（学历、经验、证书等）。 | 学历、工作年限 |  
| | `evidence` | string | **要求摘要**：具体的数值或等级要求。 | 统招本科及以上学历，英语六级（CET-6）。 |  
| **professionalism** | `name` | string | **职业素养要求**：软技能、工作态度、价值观及沟通协调能力。 | 抗压能力、团队协作 |  
| | `evidence` | string | **要求摘要**：该素质在岗位工作中的体现形式。 | 能够适应高强度项目节奏，具备跨部门沟通推动力。 |
|**paths**|`path`|string|发展路径(发展路径必须以`"-"`间隔，方便解析)|初级工程师-中级工程师-高级-架构师|  
||`requisitions`|string|走这条发展路径需要哪些资格与技能|掌握C/C++ 深度语法（C++11 及以上、STL、多线程、内存管理）、操作系统原理、TCP/IP 网络编程、Linux 开发环境（GCC、GDB、CMake、epoll）与数据结构算法；进阶需精通高并发架构、性能优化、无锁编程，熟悉 Redis 等开源组件与分布式基础|

* 3.具体流程：先在知识库中检索，如果检索不到则调用大模型生成。

### 二、知识库设计

对于知识库，我希望达到以下效果：

* 1.分片逻辑，应该是每一个岗位画像作为一个单独分片，保证信息的完整性。

* 2.对于检索逻辑，我不需要把整个岗位画像向量化作为检索用的匹配向量，而是仅对name和summary字段对应的描述，作为语义检索，匹配成功后返回整个岗位画像。

* 3.岗位画像是JSON数据，直接用于向量化不好，提取字符串内容就行。示例：

原始JSON数据
```JSON
{ "name": "C_C++", "summary": "该岗位是软件开发的核心力量，负责底层架构设计与高性能计算..." }
```
处理后得到：
`"岗位名称：C_C++。岗位综述：该岗位是软件开发的核心力量，负责底层架构设计与高性能计算..."` 这段字符串用于向量化表示岗位画像的向量空间。

* 4.知识库位于data/knowledgebase_raw，每个文件夹都对应一个岗位画像，但是`name`字段和`paths`是缺失的，需要我们手动补全。

### 三、人岗匹配专家(matching engine)重构

由于岗位画像字段发生了变化，人岗匹配专家的逻辑需要微调，人岗匹配专家需要的岗位画像不需要paths（发展路径）部分（可以在输入前剔除这个字段），其他字段都需要，也就是说，相比原先的岗位画像，修改后多了个`name`字段。

### 四、主智能体修改指南

* 1.主智能体不需要知识库检索工具了，我们删除主智能体的知识库

* 2.主智能体只需要以下两个工具（或者是skills）
    - 学生画像工具，用于获取学生画像，注意如果学生画像之前已经提取过了就不需要调用llm专家，直接从内存获取即可。
    - 人岗匹配工具，用于检索岗位画像然后得到匹配结果。也就是说，我们合并了岗位画像检索和人岗匹配的功能。

* 3.系统实现
    - 学生画像贪婪加载(eager loading)，用户一上传简历，就立即调用学生画像提取的功能，可以与主智能体的对话异步进行。
    - 对于人岗匹配工具，大模型需要输入参数:1.匹配岗位（可以是大类的也可以是具体的，具体取决于用户的需求）。2.描述，对于匹配岗位的描述性信息，用于增强向量检索的准确性。示例如下：
    ```
    匹配岗位:Java工程师。 description:用户想要从事java微服务的开发，使用SpringBoot，SpringCloud从事后端系统的构建。
    ```
    - 人岗匹配逻辑，根据大模型输入的匹配岗位参数和描述性语句，检索知识库获取匹配度高的岗位画像，然后调用人岗匹配专家做匹配分析。

* 4.提示词注入，需要以下提示词。
    - 系统提示词，说明智能体的主要工作，说明可以使用哪些工具。
    - 系统状态：1.简历文本。2.目标岗位。3.目标岗位画像。4.匹配结果。 相比之前我们删除了学生画像状态，这个状态智能体可以通过工具获取，非必要时不获取以节省上下文。
